## Context
現在のプロジェクトでは、UIコンポーネントを独自に実装している。
Tailwind CSS v4を使用しており、shadcn/uiはTailwind CSSベースのコンポーネントライブラリであるため、統合が容易である。
shadcn/uiはコンポーネントのコードをプロジェクトに直接コピーする方式のため、カスタマイズが容易で、既存のスタイルとの統合も可能。

## Goals / Non-Goals
- Goals:
  - 既存の機能を維持しつつ、shadcn/uiのコンポーネントに置き換える
  - 一貫性のあるデザインシステムを導入する
  - アクセシビリティを向上させる
  - 既存のスタイル（色、サイズ、レイアウト）を可能な限り維持する
- Non-Goals:
  - デザインの大幅な変更（機能要件は維持）
  - 新機能の追加
  - 既存のコンポーネントAPIの変更（内部実装のみ変更）

## Decisions
- Decision: shadcn/uiを使用する
  - 理由: Tailwind CSSベースで既存スタイルとの統合が容易、カスタマイズ可能、アクセシビリティが考慮されている
- Decision: エラーメッセージと成功メッセージに`Alert`コンポーネントを使用する
  - 理由: 一貫性のあるデザイン、アクセシビリティの向上、視覚的な強調
- Alternatives considered:
  - Material-UI: より重い、デザインの自由度が低い
  - Chakra UI: 良い選択肢だが、shadcn/uiの方がカスタマイズ性が高い
  - 現状維持: アクセシビリティやメンテナンス性の観点で改善の余地がある
  - `<p>`タグのまま: Alertコンポーネントの方が視覚的にもアクセシビリティ的にも優れている

## Risks / Trade-offs
- Risk: 既存のスタイルが完全に再現できない可能性
  - Mitigation: shadcn/uiのテーマ設定（`components.json`の`theme`）を調整し、必要に応じてコンポーネントをカスタマイズ
- Risk: コンポーネントの動作が変わる可能性
  - Mitigation: 各コンポーネントの置き換え後に動作確認を実施
- Trade-off: コンポーネントファイルが増える（`src/components/ui/`）
  - メリット: メンテナンス性と再利用性の向上

## Migration Plan
1. shadcn/uiの初期設定を実施
2. 基本コンポーネント（Button, Input, Textarea, Label, Alert）を追加
3. 既存コンポーネントを1つずつ置き換え、動作確認
4. エラーメッセージと成功メッセージをAlertコンポーネントに置き換え
5. スタイリングを調整
6. 全体の動作確認とテスト
7. コード品質チェック

Rollback: 各ステップでGitコミットを作成し、問題が発生した場合は前のコミットに戻す
